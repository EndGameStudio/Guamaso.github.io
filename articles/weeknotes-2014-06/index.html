Github thought/question

I work from either master or from a branch I create (typically dev). New versions or releases come as tags. I don’t see the benefit to keeping versions in their own branch. New work, fixes, or sandboxes: yes, but not old versions. This is all raw thought; I haven’t done any research on this yet, so I could be wrong. Still, when I find a repo that uses this, I would expect the most recent version to be set to master, not some version in the middle.

I found a repo that I needed for work and I noticed that the master branch seemed odd. The structure didn’t match what I was using in their commercial product. I submitted an issue without looking through their branches. If I had, I would have found that the stuff I was looking for was in a branch. I feel like there should have been a readme.md file to explain their structure and the product itself. I admit, I should have looked through the branches first. I still think it’s a weird way to structure a repo but it is on me to look before I leap. Lesson learned.

Lastly, this same repo’s owner left a comment on a prior issue that the project’s bug tracker was elsewhere. Yet, Github’s issue tracker was on. It just makes me feel like they put stuff on Github to attract attention to the product - which is perfectly fine, but if this is the business plan, ya’ gotta commit 100% (no pun intended). People should create a mental model quickly and with little effort.

I could be wrong on all of this but I doubt I am wrong on everything… just some of it. :)
