App builds

As an application grows, especially in its infancy, there needs to be some processes in place to help achieve and maintain goals for the build process.

We recently split off all our web components into their own repo and this caused all kinds of madness. It was something we couldn’t foresee and, therefore, it wasn’t something we could have done better.

Still, when something we built ends up being a blocker for current work, there needs to be some sort of AAR where we figure out what went wrong and how to avoid it in the future.

We don’t have a dev ops person… well, not explicitly anyway. I wouldn’t say we should have one, but we should remove someone from a sprint when something huge happens, something like changing our repos this extensively. This person’s sole responsibility is to monitor and maintain the build. They should document and test the build until it satisfactory. A lot of people will have a problem with this because the velocity of the next sprint will be negatively affected. It’s a tough situation. I’m curious how similar size teams handle this situation.

CommonJS

I had to familiarize myself with how CommonJS works a bit this week. I don’t mean the require or exports keywords but more how it looks for files and which thing to use when there could be multiples. Maybe I’m not looking right but I couldn’t seem to find much on this type of information.
